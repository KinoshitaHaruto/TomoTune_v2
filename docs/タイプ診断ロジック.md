# 診断ロジック設計メモ

## 問題設定

### やりたいこと

ユーザーの音楽嗜好を4軸・16タイプで分類し、いいね（ハートボタン）を押すたびにリアルタイムでスコアを更新・精度を高めていく仕組みを作る。

### 診断フロー（全体像）

```
1. 初回アンケート（Survey）
   └── 12問の質問 → 4軸スコアを初期設定

2. 楽曲へのいいね（繰り返し）
   └── SpotifyのAudio Featuresを取得
   └── 特徴量 → 4軸スコアに変換 → スコアを更新
   └── Music Typeコードを再計算

3. 結果
   └── プロフィールに 16タイプのうちの1つを表示
```

### 4軸の定義

| 軸 | High (1.0) | Low (0.0) | 意味 |
|---|---|---|---|
| V_C | V（ノリ・バイブス） | C（静けさ・内向き） | 音楽のポジティブさ・賑やかさへの傾向 |
| M_A | A（雰囲気・世界観） | M（メロディ・歌詞） | インストゥルメンタル/アンビエント志向か |
| P_R | P（情熱・激しさ） | R（落ち着き・リラックス） | エネルギー・強度への傾向 |
| H_S | H（生音・アコースティック） | S（電子音・シンセ） | 音色の有機的/電子的傾向 |

16タイプ = 各軸を二値化（≥0.5 or <0.5）して4文字のコードに（例：VMPH, CARS, ...）

---

## 現在の実装（EMA）

### アルゴリズム

指数移動平均（Exponential Moving Average）

```python
alpha = 0.03  # 学習率（固定）
new_score = prev_score * (1 - alpha) + feature_value * alpha
```

### 特徴量 → 軸のマッピング（現状）

| 軸 | 使用している特徴量 |
|---|---|
| V_C | `valence` のみ |
| M_A | `instrumentalness` のみ |
| P_R | `energy` のみ |
| H_S | `acousticness` のみ |

### 問題点

1. **1軸1特徴量の脆さ**
   例：メランコリックなダンス曲（valence低い + danceability高い）をいいねすると「C寄り（静かさ好き）」と判定されるが、実態と逆の可能性がある。

2. **不確実性が表現できない**
   いいね2回のユーザーと200回のユーザーが同スコアなら、同じ「確信度」で扱われてしまう。

3. **スコア端での硬直**
   α固定なので、スコアが0.1や0.9に近いほど変化しにくくなる。「V確定ユーザー」は何をいいねしても動かない。

4. **時系列を無視**
   1年前の好みと今の好みが同じ重みで扱われる。

## Spotify Audio Features の整理

### ⚠️ API Deprecation（2024年11月27日〜）

Spotifyは新規アプリに対してAudio Features APIを廃止した。

- **2024/11/27以前に登録・Extended Quota取得済みのアプリ** → 引き続き使える
- **それ以降に登録したアプリ** → 使えない

**実装前に必ず確認：** Spotify Developer Consoleでアプリの登録日とQuota Extensionの状態を確認すること。
使えない場合は後述の「Audio Features が使えない場合の代替案」を参照。

### 取得できる全パラメーター

#### 診断に活用できるもの

| 特徴量 | 値の範囲 | 説明 | 関連する軸 |
|---|---|---|---|
| `valence` | 0.0〜1.0 | 音楽的ポジティブさ（高→明るい、低→暗い） | V_C（主） |
| `danceability` | 0.0〜1.0 | 踊りやすさ（テンポ・リズム安定性から算出） | V_C（副） |
| `energy` | 0.0〜1.0 | 知覚的な激しさ・勢い | P_R（主） |
| `tempo` | BPM（通常60〜200） | 曲のテンポ（正規化: `(tempo-60)/140`） | P_R（副） |
| `loudness` | -60〜0 dB | 全体の音量 | P_R（副・エネルギー感の補助） |
| `instrumentalness` | 0.0〜1.0 | ボーカルの不在度（高→インスト） | M_A（主） |
| `speechiness` | 0.0〜1.0 | 話し言葉・ラップ成分の強さ | M_A（参考） |
| `acousticness` | 0.0〜1.0 | 生音らしさ（高→実楽器） | H_S（主） |
| `liveness` | 0.0〜1.0 | ライブ感・聴衆の存在度 | H_S（副） |

#### 診断への活用が難しいもの

| 特徴量 | 値の範囲 | 説明 |
|---|---|---|
| `key` | -1〜11 | 音階（-1=検出不可、0=C、1=C♯、...11=B） |
| `mode` | 0 or 1 | 0=短調、1=長調（V_C の微弱な補助にはなりうる） |
| `time_signature` | 3〜7 | 1小節の拍数 |
| `duration_ms` | ミリ秒 | 曲の長さ |

### Audio Features が使えない場合の代替案

| 代替案 | 実現性 | コスト | 内容 |
|---|---|---|---|
| **ListenBrainz API** | 高 | 中 | オープンソース。Audio Features相当の情報を提供 |
| **Last.fm API** | 中 | 中 | リスニング履歴・タグからメタデータ推定 |
| **Track Objectのみで診断** | 高 | 低 | `mode`（長調/短調）、`explicit`、アーティスト数などで粗く代替（精度は大幅低下） |
| **Extended Quota申請** | 低 | 低 | Spotifyに申請。新規アプリは承認される可能性が低い |

### 軸ごとの多特徴量化の案

`valence` と `danceability` は相関するが、必ずしも一致しない（例：憂鬱なテクノ → valence低い + danceability高い）。どちらがより「V_C軸の意図」に近いかを踏まえて重みを決める必要がある。

```
V_C = w1 * valence + w2 * danceability   （w1 + w2 = 1）
P_R = w3 * energy  + w4 * tempo_norm     （tempo_norm = (tempo - 60) / 140）
M_A = instrumentalness                   （変えない）
H_S = acousticness                       （変えない）
```

重み（w1〜w4）の決め方：
- 直感・ドメイン知識で決める（例：w1=0.6, w2=0.4）
- 複数ユーザーに「どちらのサンプルが自分の好みに近いか」を聞いてA/Bで調整する
- 将来的にはアンケート結果とaudio featuresの相関から自動推定できる可能性

## 改善手法の候補

### 案1：適応的EMA（改善コスト: 低）

```python
# α をいいね回数に応じて小さくする
alpha = alpha_0 / sqrt(n_likes)
```

- 最初は大きく動き、徐々に安定する（人間の直感と合致）
- ベイズ更新の簡易近似版
- 現在の実装への変更が最小

---

### 案2：ベイズ更新 / Beta分布（改善コスト: 中）

各軸を Beta(α, β) 分布で管理する。

```python
# 初期（アンケート後）
alpha = 5, beta = 5  → 平均0.5、不確実性大

# いいねのたびに更新
alpha += feature_value * weight
beta  += (1 - feature_value) * weight

# スコア = alpha / (alpha + beta)
# 信頼度 = alpha + beta（大きいほど確信）
```

- 不確実性が数値で持てる → 「まだ判定中」が表現できる
- 少ないいいねで大きく動き、多いいいねで安定する（自然な動作）
- DBへの変更は最小（α・β各4カラム追加）

---

### 案3：ディスカウント付きベイズ更新（改善コスト: 中）

案2に時系列減衰を追加。**履歴データは一切不要**。

```python
gamma = 0.99  # 減衰率（1いいねごとに古い記憶を少しずつ薄める）

alpha_new = gamma * alpha_old + feature_value * weight
beta_new  = gamma * beta_old  + (1 - feature_value) * weight
```

- 古いいいねの影響が自然に薄れる
- 保存するのは α と β だけ（ストレージ増加なし）
- 1回の更新は O(1) → スケーラビリティ問題なし

---

### 案4：カルマンフィルター（改善コスト: 中〜高）★本命候補

元々は宇宙船・GPSの位置追跡のアルゴリズム。「変化する真の状態をノイジーな観測から推定する」という問題に対して数学的に最適な解を与える。

**ユーザーの嗜好追跡への適用：**
- 真の状態 = ユーザーの「本当の」音楽嗜好（隠れた変数）
- 観測 = いいねした曲のaudio features（真の嗜好のノイジーな反映）

**各軸ごとに管理する値：**

```
μ  (平均)：現在の推定スコア（0.0〜1.0）
σ² (分散)：推定の不確実性
```

**更新の流れ（いいねが来るたびに）：**

```python
# --- パラメータ ---
Q = 0.001   # プロセスノイズ：好みが自然に変化する速さ
R = 0.05    # 観測ノイズ：1いいねが真の好みをどれだけ正確に反映するか

# --- 1. 時間更新（不確実性を増やす）---
# ここを「経過日数 * Q」にすると時間ベースの減衰になる
sigma2_pred = sigma2_prev + Q

# --- 2. カルマンゲイン計算 ---
K = sigma2_pred / (sigma2_pred + R)
# K が高い = 不確実なのでよく聴く / K が低い = 確信が強く動きにくい

# --- 3. 状態更新 ---
mu_new     = mu_prev + K * (observation - mu_prev)
sigma2_new = (1 - K) * sigma2_pred
```

**なぜ良いか：**
- K（カルマンゲイン）が自動でEMAのαに相当、しかも適応的に変化する
- 不確実なとき → K大きく → 新しいいいねを強く反映
- 確信があるとき → K小さく → じっくり変化
- Q の項を `Q * 経過日数` にするだけで時系列減衰が実現（履歴不要）
- 数学的に「線形ガウス系における最適推定」が保証されている

**DBへの変更：**
- `score_vc`, `score_ma`, `score_pr`, `score_hs`（既存）→ μとして流用
- `var_vc`, `var_ma`, `var_pr`, `var_hs`（追加）→ σ² として管理
- `last_liked_at`（追加）→ 経過日数の計算用

**スケーラビリティ：**
- 1回の更新: O(1) の演算
- 100ユーザー × 500いいね = 50,000回の更新でも問題なし
- ストレージ: ユーザーあたり数カラム追加のみ
